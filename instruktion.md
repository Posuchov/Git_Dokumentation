__Что такое Git и зачем он нужен?__

**Git** - это консольная утилита, для отслеживания и ведения истории изменения версий, файлов, в проекте. Является самой популярной реализацией систем контроля версий в мире. Чаще всего его используют для кода, но можно и для других файлов. Например, для картинок - полезно для дизайнеров.

С помощью Git вы можно откатить свой проект до более старой версии, сравнивать, анализировать или сливать свои изменения в репозиторий.

**Репозиторий** - это хранилище вашего кода и историй его изменений. Git может работать локально и удалённо, репозитории хранятся в определенных папках на жестком диске или, сервере или.
Так же репозитории можно хранить и в интернете, в облачном хранилище, на удалённом сервере. Обычно для этого используют сервисы:

- GitHub
- Bitbucket
- GitLab

Каждая точка сохранения вашего проекта носит название коммит (**commit**). У каждого commit-a есть hash (уникальный id) и комментарий. Из таких commit-ов собирается ветка. Ветка - это история изменений. У каждой ветки есть свое название. Репозиторий может содержать в себе несколько веток, которые создаются из других веток или вливаются в них.

**Установка**

Основой интерфейс для работы с Git является консоль/терминал. Это не совсем удобно, тем более для новичков, поэтому предлагают установить дополнительную программу с графическим интерфейсом (для лучшего визуального восприятия и удобства работы).

* Для начала, загружаем и установим сам Git.

Для операционной системы Windows: по ссылке https://git-scm.com/download/win, выбираем файл установки под разрядность ОС (32 или 64 бит), загружаем и устанавливаем.

* Для Mac OS. Открываем терминал и задаём коману:

Если установлен Homebrew (brew install git)

Если нет (git --version)

После этого появится окно, где предлогается установить Command Line Tools (CLT). Соглашаемся и ждем установки. Вместе с CLT установиться и git

* ОС Linux. Открываем терминал и задаём команду.

Для Debian или Ubuntu (sudo apt install git)

Для CentOS (sudo yum install git)

**Настройка**

Установили Git, для начала его необходимо настроить, необходимо представиться, для того чтобы при создании commit, указывался автор и его контактный адрес электронной почты.

Открываем терминал (Linux и MacOS) или консоль (Windows) и вводим следующие команды.

* Установим имя пользователя

git config --global user.name "вводим имя пользователя"

* Установим адрес электронной почты email.

git config --global user.email "адрес_почты@email.com"

Для проверки правильеости ввода и установки имени и адреса электронной почты, необходимо ввести команды:

git config user.name и git config user.email

Для полноценной работы нужен текстовый редактор, например блокнот. Но более удобна программа Visual Studio Code. Существуют версии для ОС Linux, Mac и Windows.
Программа очень удобна, её используют как редактор кода, оболочку для системы контроля версий, она пригодится во время самого программирования.
Узнать и загрузить Visual Studio Code можно по ссылке https://code.visualstudio.com/download

После установки Visual Studio Code и проверки интеграции Git Bash в профиле вашей OC Windows команда в строке поиска (def prof win), открываем терминал и начинаем работу. Для начала работы так же необходимо представиться пользователю, с помощью команд описанных выше.

# Основные команды Git 

## Репозиторий

*git init* - инициализация локального репозитория.

*git status* - просмотр состояния репозитория, получить информацию в терминале Git о текущем сосотоянии.

## Добавить файл в коммит

*git add <имя файла>* - добавление нового файла в коммит.

*git add  .* или *git add --all* - добавление всех файлов в коммит.

## Коммит

*git commit -m <сообщение к коммиту>* - создаёт коммит(сохранение). Все файлы для коммита должны быть ***ДОБАВЛЕНЫ***, изменения в файле ***СОХРАНЕНЫ***, сообщение к коммиту писать ***ОБЯЗАТЕЛЬНО***.

*git commit -am <сообщение к коммиту>* - одновременно производит добавление изменений/файлов и создаёт коммит.

## Журнал изменений

*git log* - просмотр всех сделанных изменений в репозитори, вывод на экран терминала истории всех коммитов с их полными хэш-кодами.

*git log --oneline* - более удобный и сокращённый вывод истории с начальными символами хэш-кода.

*git diff* - отображает разницу между текущим файлом и этим файлом с коммит. 

## Перемещение между сохранениями
*git checkout <номер коммита>* перемещение между коммитами, для номера коммита достаточно первых 4-5 символов хэш-кода коммита.

## Ветки в Git

### Создание ветки

*git branch <название новой ветки>* - создать новую ветку.

### Слияние веток

*git merge <название ветки>* - дабавить ветку в актуальную/текущую ветку. Команда вызывается из ветки куда добавляется другая ветка.

### Удаление веток
*git branch -d <имя ветки>* - удаление ветки, удаление дополнительной ветки только когда эта ветка полностью интегрирована в главную ветку (master) и удаление ветки не грозит потерей данных.

### Перемещение между ветками

*git checkout <имя ветки>* - перемещение между ветками.

*git checkout master* - перемещает в главную ветку с именем master.

# Синтаксис языка Markdown
## Выделение текста
Чтобы выделить текст курсивом необходимо обрамить текст (*) звёздочками или знаками нижнего подчёркивания (_). Например, *вот так* или _вот так_.

Чтобы выделить текст полужирным необходимо обрамить текст (**) или двойным знаком нижнего подчёркивания (__). Например, **вот так** или __вот так__.

Альтернативные способы выделения текста курсивом или жирным нужны для совмещения обоих способов выделения текста в тексте. Например, _текст может быть выделен курсивом и при этом быть **полужирным**_.

Чтобы выделить текст в заголовок, перед ним необходимо печатать символ (#). Количество (#, ##, ###, ####, #####, ######) задаёт уровень заголовка (величина текста заголовка обратно пропорциональна количеству #, поддерживается 6 уровней).

Чтобы выделить текст полужирным курсивом необходимо обрамить текст (***). Например ***вот так***

Чтобы отобразить зачёркнутый текст необходимо обрамить текст (~~).

Например ~~вот так~~

## Списки
Чтобы добавить ненумерованные списки, необходимо пункты списков выделить (*) или знаком (+). Например, вот так:

* Элемент 1
* Элемент 2
* Элемент 3
+ Элемент 4

Чтобы добавить нумерованные списки, необходимо пункты списков нумеровать.
Например, вот так:

1. Элемент
2. Элемент
3. Элемент
4. Элемент

## Работа с изображениями

Чтобы вставить изображение в текст, пишем строку:

**![описание файла с изображением] (имя файла)**

![konspekt_1](Konspekt_1.JPEG)

Если имя файла изображения указано корректно, в окне предварительного просмотра отображается наше изображение. Если имя файла задано с ошибкой появится только текст с  описанием файла.

![Konspekt_2](Konspekt_2.JPEG)
![Screenshot_03.06.2023](Screenshot_03.06.2023.jpg)

Не принято добавлять или сохранять непосредственно в репозитории оригиналы файлов или большие объёмы данных, в основном работают с текстовой составляющей. Исходный код программы, который так же является тексьтовым файлом добавляется в сохранение версий, а основной материал находится в отдельных папках, на отдельных носителях информации или сервере. Чтобы Git в дальнейшем не обращался и не делал запрос непосредственно о добавленных файлах, необходимо создать файл (.gitignor) и в его строке описании вносим имена тех файлов которые должны игнорироваться Git. После создания и сохранения изменений этого файла, Git игнорирует файлы указанные в списке, сами файлы в проводнике отмечаются (серым).

# Конфликт
Конфлик может возникнуть при слиянии ветки master с дополнительными ветками (черновиками), если в них существуют несовпадения в сохранённых версиях, файлах. Git при слиянии не понимает какой из двух вариантов использовать, возникает конфлик, который можно разрешить предложенными далее способами. Так же есть возможность проигнорировать конфликтное слияние или вернуться назад к рабочей версии ветки, если слияние было произведено с присутствием конфликта.

## Работа с удалёнными репозиториями


Перед тем, как скопировать репозиторий GitHub в Visual Studio Code (VSC), действительно важно убедиться, что не существует уже настроенного репозитория в VSC на вашем компьютере. Это гарантирует, что вы начинаете с чистого состояния и избегаете возможных конфликтов.

Вместо использования команды git status для проверки статуса репозитория в VSC, вы можете вместо этого либо удалить настроенный репозиторий, либо открыть другую папку.

Вот как вы можете выполнить оба действия:

Открыть другую папку: В VSC выберите "File" (Файл) в верхнем меню и затем выберите "Open Folder" (Открыть папку). Затем выберите папку, в которой вы хотите сохранить скопированный репозиторий GitHub. Это позволит открыть новую папку и начать работу с чистым состоянием.

Удалить настроенный репозиторий: Если вы уже открыли папку в VSC и настроили репозиторий, вы можете удалить его, чтобы начать с чистого состояния. Для этого в терминале VSC выполните команду git rm --cached <file> для каждого файла, которые вы хотите удалить из отслеживания. Если вы хотите удалить весь репозиторий, можно использовать команду git rm -r --cached .. После выполнения этой команды VSC больше не будет отслеживать репозиторий.

Оба варианта помогут вам начать с чистого состояния и избежать возможных проблем при копировании репозитория GitHub в VSC.
